## 1) leanNunez/Ecommerce_Tech — Baseline

### Estado general
- Proyecto **full-stack e-commerce SPA** para productos de tecnología.
- Frontend en **React 19 + TypeScript** con arquitectura **Feature-Sliced Design (FSD)**.
- Backend **Node.js + Express** ejecutado con **Bun**.
- Base de datos **PostgreSQL en Neon** vía **Prisma**.
- Deploy: **Vercel (frontend)** + **Render (backend)**.

### Features funcionales (existentes)
- Catálogo de productos con **filtros**, **búsqueda**, **sorting** y **paginación**.
- Página de detalle con galería e interacción por variantes.
- Carrito persistente.
- Wishlist.
- Checkout completo con selección de dirección.
- Historial de órdenes y detalle de orden.
- Auth:
  - registro/login
  - **JWT access + refresh** (refresh en **cookie HttpOnly**)
  - forgot/reset password
- Admin panel:
  - CRUD de productos con upload de imágenes
  - gestión de órdenes, usuarios, categorías
- Variantes de producto con upload por variante (Cloudinary).
- **RBAC** customer/admin.
- Diseño responsive.

### Arquitectura (existente)
**Frontend (FSD):**
- `src/app`, `src/pages`, `src/widgets`, `src/features`, `src/entities`, `src/shared`.

**Backend (REST):**
- `server/src/routes`, `server/src/middleware`, `server/src/lib`.

### Testing (existente)
- Frontend: **Vitest + React Testing Library**.
- Backend: **Supertest**.
- Incluye tests unitarios/componentes e integración (al menos `/auth` y `/products`) contra DB real (Neon) según README.

### Lo que explícitamente NO era parte del baseline (antes de mejoras)
- Búsqueda **semántica** (embeddings) ni vector store (pgvector/Chroma) aplicado al catálogo.
- Recomendador por similitud semántica (`/products/:id/similar`) basado en embeddings.
- Asistente de compra con **tool-calling** que interactúe con carrito/catálogo.
- Evaluación/benchmarks de relevancia para search semántica.
- Observabilidad “seria” (métricas, trazas, dashboard) más allá de logs básicos.
- E2E tests completos (Playwright/Cypress) *no confirmados* en el baseline.
- CI/CD con GitHub Actions *no confirmado* en el baseline.

## 2) leanNunez/portfolio-chatbot — Baseline

### Estado general
- Chatbot de portfolio: **RAG sobre CV/experiencia**.
- Backend: **FastAPI + Uvicorn**.
- Vector store: **ChromaDB**.
- Embeddings: Google `models/embedding-004`.
- LLM primario: **Gemini `gemini-2.0-flash`**.
- Fallback LLM: **Groq `llama-3.3-70b-versatile`** si Gemini falla o limita cuota.
- Deploy: **Hugging Face Spaces (Docker) backend** + **Vercel frontend**.

### Flujo RAG (existente)
1. User pregunta
2. Se convierte a embedding (Google AI)
3. Se buscan chunks similares en ChromaDB
4. Se construye prompt con contexto recuperado
5. Se llama al LLM y se devuelve respuesta (streaming según README)

### Knowledge base (existente)
- Fuente: archivos `.md` dentro de `backend/data/` (bio/cv/projects/skills).

### Ingest (existente)
- Script `backend/scripts/ingest.py`.
- En HF Spaces corre en cada deploy (vía `start.sh`):
  - reingesta `.md` → embeddings → ChromaDB.
- Decisión explícita: **ChromaDB efímero** (se regenera por deploy), porque el KB es estático.

### Arquitectura / Docs (existente)
- README principal con diagrama y explicación.
- Decisiones: sin LangChain por conflictos; fallback Gemini→Groq.

### Lo que explícitamente NO era parte del baseline (antes de mejoras)
- Evaluación RAG automatizada (dataset, métricas hit@k/MRR/groundedness, CI gating).
- Observabilidad avanzada (métricas, dashboard, cost/tokens estimados por request).
- Rate limiting/anti-abuse *no confirmado* en el baseline.
- Políticas de PII/log retention formalizadas.
- “Assistant” con tools/actions sobre sistemas externos (más allá de Q&A sobre KB).
